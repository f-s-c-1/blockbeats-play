# 草原杯 H5 现场互动系统 · 产品需求文档（PRD）

> 一句话定位：**管理员（主持人）当导演，参与者手机当举牌道具**。
> 全场处于「一个房间、一个当前环节」，参与者屏幕永远是控制台的投影——
> 管理员不推，参与者就停在等待页；管理员一推，对应的人按各自「队伍 + 隐藏身份」亮起对应内容。
>
> 目标：把团建手册附录里的纸质道具（词条、积分榜、抽签卡、内鬼身份牌、任务卡、计时器）全部数字化，**少带道具、强可控**。

---

## 0. 名词约定

| 名词 | 含义 |
|---|---|
| **房间 Room** | 一场团建一个房间，有唯一房间码（可输入，避免多场冲突）。全场状态都挂在房间上。 |
| **管理员 Admin** | 主持人，上帝视角，唯一能推进节奏的人。可备用 1 台同时在线。 |
| **参与者 Player** | 约 40 人，起名加入，被动接收。 |
| **当前环节 currentStage** | 全场同一时刻只有一个。决定参与者屏幕显示哪一类界面。 |
| **队伍 Team** | 抽签后 4–5 队，每人固定归属一队。 |
| **隐藏身份 secretRole** | `normal`（普通）/ `spy`（草原内鬼）。仅本人 + 管理员可见。 |
| **推送 push** | 管理员把某环节内容下发到对应参与者屏幕的动作。不推则不显示。 |

## 1. 角色与权限

| 能力 | 管理员 | 参与者 |
|---|:--:|:--:|
| 创建/进入房间 | ✅ 创建 | ✅ 凭链接进入 |
| 看到全部成员名单 | ✅ | ❌（仅自己 + 同队） |
| 看到所有队的词/答案 | ✅ | ❌ |
| 看到谁是内鬼 | ✅ | ❌（仅内鬼知道自己） |
| 推进环节 / 推送内容 | ✅ | ❌ |
| 记分 / 翻倍 | ✅ | ❌（只能看积分榜） |
| 发言 | 单向广播（公告） | **仅能私信管理员**（见 §6） |
| 强制全体回等待页 | ✅ | ❌ |

## 2. 可见性三层模型（系统灵魂）

参与者**看到什么**，由三个条件叠加计算：

```
可见内容 = f( 当前环节 currentStage , 我的队伍 teamId , 我的隐藏身份 secretRole )
```

1. **当前环节** → 决定显示哪一类界面（没推任何环节 = 等待页）。
2. **队伍归属** → 只看本队信息，看不到别队。
3. **隐藏身份** → 同一环节里，内鬼比普通人多一层（隐藏身份 + 内鬼任务）。

> 核心铁律：**参与者端不主动渲染任何游戏内容**，永远默认停在「⏳ 等待主持人」。只有收到管理员 push 的、且「命中自己可见性」的内容才点亮。

## 3. 环节 × 角色 × 可见内容 对照表（全量 PRD 核心）

> 列说明：
> - **可见性类型**：A 私密发牌（每人不同）/ B 单人可见（仅指定人看）/ C 全员同屏（都一样）/ D 投票（都能投看不到票数）/ E 身份分发（按队+身份）/ **F 个人存活淘汰（人人有独立存活状态，实时全员可见）**
> - **普通参与者所见** / **内鬼额外所见** / **管理员所见（上帝视角）**

### 3.1 进入与分组

| 环节 | 类型 | 普通参与者所见 | 内鬼额外所见 | 管理员所见 / 操作 |

| **登录起名** | — | 起名输入框 → 提交后「✅ 已加入，等待主持人」 | 同左 | 实时名单「已加入 38/40」、谁在线 |

| **队名/队长设定** | E | 本队可填队名（队长操作），显示本队队名+队长 | 同左 | 看到所有队名，回填积分榜 |

## 4. 房间状态机（后端核心数据模型）

```
Room {
  code            string        // 房间码，可输入，多场隔离
  phase           "lobby" | "running" | "ended"
  currentStage    StageState | null   // 当前环节，null = 全体等待页
  members         Player[]
  teams           Team[]
  scoreMultiplier number        // 默认 1，最后一天置 2
  createdAt
}

Player {
  id              string        // 设备/会话标识
  name            string        // 自己起的名字
  avatar          string        // 选中的 emoji（如 "🐺"），仅存字符，零上传零存储
  teamId          string | null
  secretRole      "normal" | "spy"
  online          boolean
  // 派生：该 player 的可见内容由 currentStage + teamId + secretRole 计算
  // 渲染：头像 = avatar(emoji) + 名字 + 队伍色，三者组合辨识，不强制 emoji 唯一
}

Team {
  id, name, captainId, score
}

StageState {            // 不同游戏 type 携带不同 payload
  type            "lobby" | "draw" | "undercover" | "charades"
                  | "vote" | "scoreboard" | "timer" | "task" | "reveal"
                  | "lastman" | ...                       // lastman = 个人淘汰赛（拽尾巴）
  visibility      "A" | "B" | "C" | "D" | "E" | "F"
  payload         object        // 例：undercover→{assignment}; charades→{actorId,word};
                                //     lastman→{ alive:{playerId->bool}, aliveCount, championId? }
  startedAt
}
```

> **F 类的存活状态故意挂在 `payload.alive` 上，不挂 Player。** 队伍/内鬼是全程属性，存活是单局属性——换个游戏就该清零。挂 payload 后切环节自动失效，不污染 Player，符合状态机干净原则。开赛时 `alive` 初始化为所有参与者 = true。

**可见性计算（参与者拉取 / 推送时执行）：**
```
getVisibleView(player, currentStage):
  if currentStage == null:        return WAITING_VIEW
  switch currentStage.visibility:
    C: return currentStage.payload                       // 全员同屏
    B: return player.id == payload.actorId ? {word} : {guessHint}
    A: return { myWord: payload.assignment[player.id] }  // 每人查自己
    D: return player.hasVoted ? VOTED : { candidates }
    E: base = teamView(player.teamId, payload)
       if player.secretRole == "spy": base += spyTaskView // 内鬼多一层
       return base
    F: return { alive: payload.alive[player.id],         // 我是否存活
                aliveCount: payload.aliveCount,          // 全场剩几人（全员可见）
                champion: payload.championId }           // 冠军产生后揭晓
```

> 状态机决定了：移植到 Nuxt + 轻后端时，「服务端只维护一个 Room 状态 + 一套 getVisibleView」，前端按角色渲染即可。当前 HTML 原型的轮询可替换为 WebSocket / SSE 推送。

## 5. 管理员控制台 = 节奏遥控器（交互）

五个区域：

1. **环节切换区**：当晚要玩的游戏一排按钮，点哪个进入哪个配置；全场同一刻只一个 currentStage。
2. **配置区**：选词对 / 选参与者 / 指定比划人 / 设时长 → 配好点「推送」才下发。
3. **上帝视角监控区**：每人当前屏幕停在哪、谁是内鬼（高亮）、谁没投票、各队分。
4. **积分操作区**：每队 ±，「分值翻倍」开关。
5. **强制同步**：「全体回等待页」按钮，随时拉回 `currentStage = null` 防乱。

每个按钮点下去的前后端动作（线框逻辑）在 §7 细化。

## 6. 参与者 → 管理员 单向私信（你确认的「对话」形态）

**不是聊天室，是一条上行通道**，把三种现场动作收口：

| 用途 | 参与者操作 | 管理员所见 |
|---|---|---|
| 求助 / 举手 | 点「🙋 找主持人」+ 一句话 | 收件箱：谁、哪队、说了啥 |
| 提交答案 | 某些环节里直接提交文字答案 | 按环节归类显示 |
| 偷偷举报内鬼 | 「我怀疑 XXX」 | 仅管理员可见，可作揭晓彩蛋 |

约束：
- 参与者**只能发给管理员**，收不到别的参与者消息，**无横向聊天**（杜绝刷屏）。
- 管理员可单向「广播公告」回全场（§3.5 公告环节），但不点名回私信，避免变群聊。
- 频率限制：每人 N 秒一条，防刷。

**上行通道开关策略（已确认）：默认关，按需开。**
- 默认 `room.uplinkOpen == false`，参与者看不到「找主持人」入口，杜绝全程被 40 人私信轰炸。
- 仅两类场景由管理员手动 `admin:toggleUplink` 打开：
  1. **内鬼揭晓前**（§3.5 内鬼投票/揭晓）→ 开放「🕵️ 偷偷举报内鬼」。
  2. **需要收答案的环节**（如默契大考验提交答案）→ 开放「提交答案」。
- 这两类环节由后端打标 `requiresUplink: true`；管理员推送该环节时，控制台**提示**「建议开启上行通道」（不自动开，仍由管理员点，保持节奏掌控）。
- 环节切换或 `stage:clear` 时，通道**自动复位为关**，防止开了忘关。

> 复杂度评估：本质是 `messages[] { fromPlayerId, text, ts, stageContext }` 单表 + 管理员收件箱视图，**比全功能 IM 低一个数量级**，且天然喂给上帝视角。

## 7. 控制台按钮 → 前后端动作（完整线框）

> 这是写 Nuxt 代码前的最后一张图纸。每条按统一 7 步骨架描述：
> **① 触发** 谁点 / 前置条件 → **② 前端** 控制台本地动作 → **③ 请求** 发什么（事件名 + 载荷）→
> **④ 后端校验** → **⑤ 状态变更** 改 Room 的哪些字段 → **⑥ 推送范围** 服务端把变更推给谁 →
> **⑦ 各端渲染** 管理员屏 / 参与者屏分别变成什么。
>
> 约定：
> - 服务端是**单一可信源**。所有状态变更只在后端发生，前端永不本地猜测结果。
> - 推送统一走一个事件 `room:state`，载荷是「**针对该连接计算过的视图**」——即服务端对每个连接跑一遍 `getVisibleView(player, currentStage)` 后只发它该看的部分（参与者绝不会在网络层拿到别队/别人的词）。管理员连接收到的是**完整 Room 状态**（上帝视角）。
> - 幂等：所有写操作带 `actionId`，重复请求（断线重发）不产生副作用。

---

### 7.0 通用：参与者上行私信 `msg:send`（贯穿全程）
- **① 触发**：参与者点「🙋 找主持人」/ 提交答案 / 举报内鬼；前置 `room.uplinkOpen == true`（管理员可关，平时关、揭晓前开）。
- **② 前端**：弹一行输入，提交即禁用按钮进入 N 秒冷却。
- **③ 请求**：`msg:send { text, stageContext, actionId }`；发送者身份由 WebSocket session 绑定，客户端不自报。
- **④ 后端校验**：上行通道开？连接身份是参与者？距上次 ≥ N 秒？text 非空且 ≤ 200 字？
- **⑤ 状态变更**：`messages.push({ id, fromPlayerId, teamId, text, ts, stageContext })`。
- **⑥ 推送范围**：仅推**管理员连接**（参与者之间永不互发）。
- **⑦ 各端渲染**：管理员收件箱新增一条（带姓名/队/环节）；参与者显示「已送达，等待主持人」。

---

### 7.1 创建房间 `room:create`
- **① 触发**：管理员落地页点「创建房间」，可自定义房间码（留空则后端生成）。
- **② 前端**：管理员选「我是管理员」→ 提交房间码。
- **③ 请求**：`room:create { code?, adminName, actionId }`。
- **④ 后端校验**：code 未被占用（多场隔离）；同一 code 已存在则拒绝并提示「房间码已被占用」。
- **⑤ 状态变更**：新建 `Room { code, phase:"lobby", currentStage:null, members:[], teams:[], scoreMultiplier:1, uplinkOpen:false }`，记录 adminId。
- **⑥ 推送范围**：仅管理员本人。
- **⑦ 各端渲染**：管理员进入控制台，顶部显示房间码 + 可复制的参与者链接（`/r/<code>`）。参与者此时还没进来。

### 7.2 参与者加入 `player:join`（登录起名 + 选头像）
- **① 触发**：参与者打开 `/r/<code>` → 起名页。
- **② 前端**：输入名字 + **从一排草原主题 emoji 里点选一个当头像**（🐺🦌🐎🐑🦅🐫🌟🔥⛺🏹🍶…，默认随机选中一个，不点也能过）→ 提交。
- **③ 请求**：`player:join { code, name, avatar, clientId, actionId }`（`clientId` 存 localStorage，用于断线重连）。
- **④ 后端校验**：房间存在且 `phase != "ended"`；`clientId` 已存在 → 视为**重连**而非新建（见 7.13）；同名追加序号；avatar 缺省则后端随机分配一个 emoji（**不校验唯一**，撞了无所谓）。
- **⑤ 状态变更**：`members.push(Player{ id, name, avatar, teamId:null, secretRole:"normal", online:true })`。
- **⑥ 推送范围**：该参与者（确认加入）+ 管理员（名单刷新）。
- **⑦ 各端渲染**：参与者 → 等待页「✅ 已加入，等待主持人」（带自己的 emoji）；管理员监控区「已加入 38/40」实时 +1，每人显示为「emoji + 名字 + 队伍色」。

### 7.3 开始抽签 `draw:generate`（后台生成分队，先不下发）
> ⚠️ **分队 ≠ 内鬼指定，两步分开**（修正：手册要求内鬼是「提前通过气、愿意整活的人」，不能纯随机，故内鬼改为管理员手动指定，见 7.3.5）。
- **① 触发**：管理员点「开始抽签」，可设队数（4/5）、微平衡开关。**不再在此设内鬼**。
- **② 前端**：弹配置 → 确认。
- **③ 请求**：`draw:generate { teamCount, balance, actionId }`。
- **④ 后端校验**：`phase == "lobby"`；在线人数 ≥ 队数。
- **⑤ 状态变更**：按蛇形/随机写每个 `Player.teamId`；建 `teams[]`（暂无队名）。**currentStage 仍为 null**，只存后端不下发。
- **⑥ 推送范围**：仅管理员（预览分队 + 重抽）。
- **⑦ 各端渲染**：管理员看到完整分队预览（可「重抽」）；**参与者屏幕无变化，仍在等待页**。

### 7.3.5 手动指定内鬼 `spy:assign`（管理员挑人，非随机）
- **① 触发**：管理员在名单里**手动勾选 2–3 个内鬼**（挑事先通过气、玩得开的人）。可随时改、可在揭晓前调整。
- **② 前端**：名单多选，已选高亮。
- **③ 请求**：`spy:assign { playerIds:[...], actionId }`（全量覆盖式，传完整内鬼名单）。
- **④ 后端校验**：playerIds 都在房；数量建议 2–3（超出给 warning 不阻断）。
- **⑤ 状态变更**：先把所有人 `secretRole` 重置为 `normal`，再把 `playerIds` 置 `spy`；可同时给每个内鬼写 `spyTask`（任务卡文案）。**不下发**。
- **⑥ 推送范围**：仅管理员（内鬼名单高亮）。
- **⑦ 各端渲染**：管理员看到谁是内鬼；参与者无变化。
- **说明**：与分队解耦——可以先分队，内鬼晚点再定；身份在揭晓(7.4)时才下发到本人。

### 7.4 揭晓分组 `draw:reveal`（E 类下发）
- **① 触发**：管理员对分队预览满意 → 点「揭晓分组」。
- **② 前端**：确认。
- **③ 请求**：`stage:set { type:"draw", actionId }`。
- **④ 后端校验**：已执行过 `draw:generate`（存在 teamId 分配）。
- **⑤ 状态变更**：`currentStage = { type:"draw", visibility:"E", payload:{} }`。
- **⑥ 推送范围**：**全体**，但每连接跑 `getVisibleView`：
  - 普通参与者 → `{ team:{name,members}, role:"normal" }`（仅本队）
  - 内鬼参与者 → 上述 + `{ secretRole:"spy", spyTask }`
  - 管理员 → 完整。
- **⑦ 各端渲染**：普通人「🎴 你属于【苍狼队】，队友 A/B/C」；内鬼额外弹「🤫 你是草原内鬼，任务 ____（保密）」；管理员不变。

### 7.5 队名/队长设定 `team:setName`
- **① 触发**：队长在自己手机填本队队名（或管理员代填）。
- **② 前端**：本队成员里仅队长可见输入框。
- **③ 请求**：`team:setName { teamId, name, actionId }`。
- **④ 后端校验**：发起者属于该队（或是管理员）。
- **⑤ 状态变更**：`team.name = name`。
- **⑥ 推送范围**：该队成员 + 管理员（积分榜回填）。
- **⑦ 各端渲染**：本队显示队名；管理员积分榜出现该队名。

### 7.6 谁是卧底 `undercover:push`（A 类·私密发牌）
- **① 触发**：管理员选词对（如 烤全羊/烤羊腿）、选参与人、设几张卧底词。
- **② 前端**：词库下拉 + 勾选参与者。
- **③ 请求**：`undercover:push { wordPairId, participantIds, spyWordCount, actionId }`。
- **④ 后端校验**：参与者都在房；词对存在。
- **⑤ 状态变更**：后端**就地随机**生成 `assignment{ playerId -> word }`（决定谁拿平民词/卧底词），写入 `currentStage = { type:"undercover", visibility:"A", payload:{ assignment } }`。**assignment 整体只存后端**。
- **⑥ 推送范围**：每个参与者只收到 `{ myWord: assignment[me] }`；非参与者收到「本环节你不参与」；管理员收到全表（谁平民/谁卧底）。
- **⑦ 各端渲染**：参与者「你的词：烤全羊」；管理员一张「谁拿什么词」明细 + 出局记录按钮。

### 7.7 你比我猜 / 你画我猜 `charades:push`（B 类·单人可见）
- **① 触发**：管理员指定本轮比划者、选词。
- **② 前端**：选比划者 + 词。
- **③ 请求**：`charades:push { actorId, word, durationSec, actionId }`。
- **④ 后端校验**：actor 在房。
- **⑤ 状态变更**：`currentStage = { type:"charades", visibility:"B", payload:{ actorId, word, durationSec } }`。
- **⑥ 推送范围**：比划者收 `{ word }`；其他人收 `{ guessHint:"👀 猜！" }`；管理员收 `{ word }` + 计时控制。
- **⑦ 各端渲染**：比划者屏「套马杆」；其他人「👀 猜！」；管理员看词 + 倒计时 + 喊过/跳过。（喊过/跳过见 7.10 计时控制族。）

### 7.8 全员同屏类 `stage:set`（C 类：积分榜 / 倒计时 / 公告 / 规则卡 / 定格 / 计数 / 揭晓展示）
- **① 触发**：管理员点对应按钮（推积分榜 / 推规则卡 / 推公告…）。
- **② 前端**：积分榜/规则卡直接推；公告需输入文字；计数类附带「+1」。
- **③ 请求**：`stage:set { type, visibility:"C", payload, actionId }`，例：
  - 积分榜 `{ type:"scoreboard", payload:{} }`（分数从 teams 实时取）
  - 公告 `{ type:"announce", payload:{ text } }`
  - 计数 `{ type:"counter", payload:{ count } }`
- **④ 后端校验**：payload 合法（公告非空等）。
- **⑤ 状态变更**：`currentStage = {...}`；积分榜不冗余存分，渲染时读 `teams.score × scoreMultiplier`。
- **⑥ 推送范围**：全体 + 管理员，所有人**同一份** payload。
- **⑦ 各端渲染**：全员同屏一致（排名 / 公告横幅 / 规则卡 / 大字定格 / 计数）。

### 7.9 倒计时 `timer:*`（C 类·独立控制族）
- **① 触发**：管理员设时长 → 开始/暂停/清零。
- **② 前端**：输入秒数 + 三个控制按钮。
- **③ 请求**：`timer:start { durationSec }` / `timer:pause` / `timer:reset`。
- **④ 后端校验**：durationSec > 0。
- **⑤ 状态变更**：后端记 `currentStage.payload.timer = { endsAt, paused, remaining }`（存**绝对结束时间戳**，前端按本地时钟倒推，避免逐秒推送）。
- **⑥ 推送范围**：全体（只在 start/pause/reset 时推一次，不每秒推）。
- **⑦ 各端渲染**：所有屏按 `endsAt` 各自跑秒，天然同步；暂停时显示 remaining。

### 7.10 计时/判定控制族（喊过 / 跳过 / 出局 / 计数+1 / 抢答判定）
- **① 触发**：依附于当前游戏环节（charades 的过/跳、undercover 的出局、counter 的 +1、抢唱的判先后）。
- **② 前端**：管理员当前环节面板上的小操作按钮。
- **③ 请求**：`stage:action { kind, targetId?, actionId }`。
- **④ 后端校验**：当前 currentStage 类型支持该 action。
- **⑤ 状态变更**：更新对应 payload（如 `counter.count += 1`、记录出局者、记抢答顺序）。
- **⑥ 推送范围**：按可见性——counter 全员可见则全推；undercover 出局信息推全员（公开出局）。
- **⑦ 各端渲染**：对应数字/状态变化。

### 7.10.5 拽尾巴 · 个人淘汰赛 `lastman:*`（F 类·个人存活淘汰）
> 线下满场跑、互拽尾巴；屏幕只当**实时存活看板**——谁被拽掉了管理员点一下，全场屏幕同步。

- **开局 `lastman:start`**
  - **①** 管理员点「开始拽尾巴」，可选参赛范围（全员 / 指定队 / 指定人）。
  - **③** `lastman:start { participantIds?, actionId }`（不传=全员）。
  - **④** 参赛者 ≥ 2 人。
  - **⑤** `currentStage = { type:"lastman", visibility:"F", payload:{ alive:{每个参赛者->true}, aliveCount:N, championId:null } }`。
  - **⑥⑦** 全员屏：自己「🟢 存活」+「剩 N 人」；管理员：全员存活网格（绿=活/灰=淘汰），每格可点。
- **淘汰某人 `lastman:eliminate`**
  - **①** 谁被拽掉，管理员点他的格子。
  - **③** `lastman:eliminate { targetId, actionId }`。
  - **④** target 当前存活。
  - **⑤** `alive[target]=false`；`aliveCount--`；若 `aliveCount==1` → 自动把剩下那人写入 `championId`。
  - **⑥⑦** 被淘汰者屏翻「💀 已淘汰」；全员「剩 N 人」实时 -1；管理员该格变灰。
- **误点复活 `lastman:revive`**
  - **③** `lastman:revive { targetId, actionId }`（防手滑，管理员专用）。
  - **⑤** `alive[target]=true`；`aliveCount++`；若曾产生 championId 则清空。
  - **⑥⑦** 对应回滚。
- **结算 `lastman:finish`**
  - **①** 产生冠军（或管理员手动定「最后 N 人」）时点结算。
  - **⑤** 锁定 `championId`，可附「给冠军所在队 +X 分」直接调 §7.12 `score:adjust`。
  - **⑥⑦** 全员屏：「🏆 冠军是 XXX」；管理员可一键记分并切回积分榜。
- **说明**：内鬼在此环节无特权，正常参赛被拽就淘汰。存活状态随 `stage:clear` / 切环节自动失效（挂 payload，不挂 Player）。

### 7.11 内鬼投票 `vote:*`（D 类）
- **开投票 `vote:open`**
  - **③** `vote:open { candidateIds?, actionId }`（默认全员为候选）。
  - **⑤** `currentStage = { type:"vote", visibility:"D", payload:{ candidates, ballots:{} } }`。
  - **⑥⑦** 全员屏出现单选名单；管理员看到实时票数分布（参与者看不到）。
- **参与者投票 `vote:cast`**
  - **③** `vote:cast { targetId, actionId }`；投票人由 WebSocket session 绑定，客户端不自报。
  - **④** 该连接是参与者，target 在候选人内。
  - **⑤** `ballots[session.playerId] = targetId`。
  - **⑥⑦** 投票者屏变「已投票，等待揭晓」（不显示票数）；管理员票数实时更新。
- **公布票数 `vote:revealCount`**
  - **⑤⑦** currentStage 切到展示票数排行（C 类），全员看到得票榜。
- **揭晓内鬼 `vote:revealSpy`**
  - **⑤** 读取后端 `secretRole=="spy"` 名单 → 推 reveal payload。
  - **⑦** 全员翻牌「真正内鬼是 XXX」；可附 7.0 收到的举报私信作彩蛋。

### 7.12 记分 / 翻倍 `score:*`
- **① 触发**：任意时刻管理员对某队 ± 分；最后一天开「分值翻倍」。
- **③ 请求**：`score:adjust { teamId, delta, actionId }` / `score:multiplier { value:2|1 }`。
- **④ 后端校验**：teamId 存在；value ∈ {1,2}。
- **⑤ 状态变更**：`team.score += delta` / `room.scoreMultiplier = value`。
- **⑥ 推送范围**：若积分榜正在屏（currentStage.type=="scoreboard"）→ 全员实时同步；否则只更新管理员视图。
- **⑦ 各端渲染**：积分榜排名实时变；翻倍状态在管理员区高亮提示。

### 7.13 强制全体回等待页 `stage:clear`
- **① 触发**：管理员点「全体回等待页」（防冷场/防乱）。
- **③ 请求**：`stage:clear { actionId }`。
- **⑤ 状态变更**：`currentStage = null`（已写入的 teamId/secretRole/score **不清除**，只清当前环节）。
- **⑥ 推送范围**：全体。
- **⑦ 各端渲染**：所有参与者瞬间回到「⏳ 等待主持人」；管理员控制台回到环节选择区。

### 7.14 断线重连 `player:rejoin`
- **① 触发**：参与者刷新/掉线后重开链接。
- **② 前端**：localStorage 取 `clientId` 自动带上，无需重新起名。
- **③ 请求**：`player:rejoin { code, clientId, actionId }`。
- **④ 后端校验**：`clientId` 命中已有 Player。
- **⑤ 状态变更**：`player.online = true`（身份/队伍/内鬼/积分全部保留）。
- **⑥ 推送范围**：该参与者补发当前 `getVisibleView`（立刻回到正确画面，含当前环节）；管理员名单标记在线。
- **⑦ 各端渲染**：参与者无缝回到「当前该看到的内容」，不丢身份。

### 7.15 备用管理员（双 Admin）
- 第二台管理员用同房间码 + 管理员口令进入，**只读完整状态 + 可接管操作**。所有写操作仍走后端单一可信源，两台管理员看到的上帝视角一致（同样靠 `room:state` 全量推送）。冲突由后端 `actionId` 串行化处理，后到的同类操作覆盖前一个。
- **分工协作模式**：双管理员不一定是「主+备」，可一个主持、一个专门点屏幕记分/点淘汰（40 人现场一个人手不够）。两台都是完整控制台，互不锁定。

### 7.17 管理员踢人 / 改名 `admin:kick` / `admin:rename`（防乱入治理）
- **① 触发**：发现路人乱入、重复加入、恶意起名。
- **③ 请求**：`admin:kick { playerId, actionId }` / `admin:rename { playerId, newName, actionId }`。
- **④ 后端校验**：发起者是管理员；playerId 在房。
- **⑤ 状态变更**：kick → 标记 `player.kicked=true` 并断开其连接（该 clientId 再 join 被拒）；rename → 改 `player.name`。
- **⑥ 推送范围**：被踢者收到「已被移出房间」；管理员名单刷新。
- **⑦ 各端渲染**：被踢者回到起名页且提示被移出；管理员列表更新。

### 7.18 管理员重连 `admin:rejoin`（节奏源不能断）
> 管理员是唯一节奏源，他掉线/锁屏/切到微信全场卡死，比参与者断线严重得多。
- **① 触发**：管理员手机锁屏、切后台、断网后回来。
- **② 前端**：控制台页常驻防锁屏提示（建议保持亮屏 / 用平板）；localStorage 存 `adminToken`。
- **③ 请求**：`admin:rejoin { code, adminToken, actionId }`。
- **④ 后端校验**：adminToken 匹配该房管理员。
- **⑤ 状态变更**：无（状态在后端，未丢）。
- **⑥ 推送范围**：补发完整 Room 状态。
- **⑦ 各端渲染**：管理员秒回到掉线前的控制台现场（当前环节、积分、内鬼名单全在）。**前提是后端做了持久化（见 §9）**。

## 9. 现场鲁棒性 & 体验设计（团建现场专项）

> 这些不是功能，是「真到草原上 40 人跑起来不翻车」的工程约束，写代码时必须内建。

### 9.1 网络兜底 + 持久化（🔴 最高优先级）
现场（火山/草原）信号差是必然，系统强依赖实时连接，必须假设「随时有人断、后端可能重启」。

- **状态持久化**：Room 全量状态（成员/队伍/内鬼/积分/当前环节）**不能只在内存**，需落地到持久层（Redis / SQLite / 文件均可）。后端进程重启后能从持久层恢复，否则身份/积分全丢——这是 §7.18 管理员重连能生效的前提。
- **跨天存活**：抽签在 Day1 大巴、内鬼揭晓在 Day3 晚，中间隔近 3 天。房间状态必须能**跨天保留**（持久化 + 房间不自动过期，或设很长 TTL）。中途若关服重开，靠持久层恢复内鬼身份。
- **断线分级**：参与者断线→7.14 重连补发；管理员断线→7.18 重连（节奏源，最关键）。
- **弱网降级**：屏幕定位为**辅助看板**，线下游戏（拽尾巴/气球/投布鲁）本来就线下玩。明确告知主持人——**关键积分主持人用纸笔备份一份**，H5 崩了团建照常进行，不把一晚压在一个网页上。
- **写操作幂等**：所有动作带 `actionId`，断线重发不重复扣分/重复淘汰（§7 已约定）。

### 9.2 防乱入治理（🔴）
链接公开转发，任何人点开都能进，必须能治理。

- **人数上限**：房间设 `maxPlayers`（如 50），满了拒绝新加入。
- **可选入场口令**：管理员可给房间设一个简单口令（如 `caoyuan`），参与者加入时输入；不设则任何人凭链接进（小规模信任场景够用）。
- **管理员治理**：踢人 / 改名 / 封禁（§7.17）。被踢的 clientId 再 join 被拒。
- **起名约束**：名字长度限制、敏感词过滤（可简单做）。

### 9.3 体验问题（🟡 影响顺不顺，不影响能不能跑）

- **谁是卧底的游戏流程边界**：7.6 只「发词」。完整玩法（轮流描述→投票出局）中，**描述靠口头，出局投票可复用 7.11 投票机制**（管理员开一轮针对本局参与者的投票）。屏幕只负责发词 + 可选的出局投票，不自动判定卧底胜负——这条边界要在 UI 上写清楚，避免主持人以为系统全自动。
- **积分翻倍语义**：放弃全局 `scoreMultiplier` 开关（容易忘关导致全乱）。改为**每次记分时手动选 ×1 / ×2**：`score:adjust { teamId, delta, multiplier }`，翻倍只作用于当次。最后一天主持人记分时点 ×2 即可。**（这会取代 §7.12 的全局 multiplier，待移植时落实）**
- **常驻信息 overlay（✅ MVP 内建，已定）**：积分榜/倒计时/公告这类**常驻信息**不和主环节互斥切换，做成可叠加的 **overlay 层**。架构升级为：
  - `currentStage`：单一主环节（抽签/谁是卧底/拽尾巴…），互斥切换，铁律不变。
  - `overlays: { timer?, announce?, scoreboard? }`：独立挂在 Room 上，可同时存在，不随主环节切换清除（除非管理员主动关）。
  - `getVisibleView` 算两层：先算主环节可见内容，再叠加 overlays（overlay 多为 C 类全员同屏，直接拼上）。
  - 例：主环节「谁是卧底」发词中，顶部仍挂一条倒计时 overlay。
- **推送提醒**：参与者长时间停等待页会收手机走神。关键环节推送时给**震动 + 提示音**（`navigator.vibrate` + 短音效），唤起注意。
- **场次分组**：控制台按钮别堆成一长条。按行程分组预设：「Day1 大巴 / Day1 晚餐 / Day3 草原 / Day3 晚餐」，每组列该时段要玩的环节，减少现场翻找。

### 9.4 取舍声明（明确不做的）
- 不做参与者横向聊天（§6 已定，只上行私信）。
- 不做真人头像上传（用 emoji，§7.2）。
- 不做扫码（用链接）。
- overlay / 场次分组 / 震动提醒列为**体验增强**，MVP 可后置，但持久化和防乱入是 MVP 必须。
